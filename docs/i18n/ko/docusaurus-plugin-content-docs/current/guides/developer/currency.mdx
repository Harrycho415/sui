Coin Registry 시스템은 `sui::coin_registry` 모듈을 통해 화폐 관리에 대한 중앙화된 접근 방식을 제공한다. 이 레지스트리는 모든 등록된 코인 타입의 메타데이터, 공급 정보, 규제 상태를 저장하는 address `0xc`에 위치한 shared object이다.

`sui::token` 모듈은 네트워크에서 토큰 생성을 처리한다. 이러한 기능에 대한 더 많은 정보는 [Closed-Loop Token](/standards/closed-loop-token.mdx) 및 [Currency](/standards/currency.mdx) Standards 문서를 참조한다.

### Currency creation process

레지스트리 시스템은 두 가지 화폐 생성 방법을 지원한다:

1. **Standard creation:** 패키지의 `init` 함수 외부에서 코인을 생성할 때 `new_currency<T>()`를 호출한다.
1. **OTW creation:** 고유성 증명을 위해 One-Time Witness와 함께 `new_currency_with_otw<T>()`를 사용한다.

두 방법 모두 확정 전에 구성을 허용하는 `CurrencyInitializer<T>`를 반환한다:

초기화 프로세스는 다음을 허용한다:

- **Supply model selection:** 고정, 소각 전용 또는 유연한 공급을 선택한다.
- **Regulatory configuration:** 필요한 경우 deny list capabilities를 추가한다.

:::caution Important

OTW 방법을 사용한 화폐 초기화 후에는 Coin Registry가 추적할 수 있는 shared `Currency` object를 생성하기 위해 `finalize_registration`을 반드시 호출해야 한다.

:::

## `DenyList`

The Sui framework는 `DenyCapV2`의 보유자가 Sui core type을 사용할 수 없는 address 목록을 지정하기 위해 접근할 수 있는 `DenyList` singleton shared object를 제공한다. 그러나 `DenyList`의 초기 사용 사례는 지정된 타입의 코인에 대한 접근을 제한하는 데 중점을 둔다. 이는 예를 들어 Sui에서 특정 address가 해당 코인을 transaction의 입력으로 사용하지 못하도록 차단하는 ability가 필요한 regulated coin을 생성할 때 유용하다. Sui의 regulated coin은 알려진 악의적 행위자가 해당 코인에 접근하지 못하도록 하는 ability를 요구하는 모든 규제를 충족한다.

:::info

`DenyList` object는 address `0x403`을 가지는 system object이다. 이를 직접 생성할 수는 없다.

:::

## Create tokens

토큰은 `sui::coin` 모듈에 정의된 `TreasuryCap`을 재사용하며 따라서 동일한 초기화 프로세스를 가진다. `coin::create_currency` 함수는 `TreasuryCap`의 고유성을 보장하고 `CoinMetadata` object의 생성을 강제한다.

Coin-like 함수는 토큰의 민팅과 소각을 수행한다. 둘 다 `TreasuryCap`이 필요하다:

- `token::mint`: 토큰을 민트한다
- `token::burn`: 토큰을 소각한다

토큰 작업의 전체 세부 사항은 [Closed-Loop Token](../../standards/closed-loop-token.mdx) standard를 참조한다.
