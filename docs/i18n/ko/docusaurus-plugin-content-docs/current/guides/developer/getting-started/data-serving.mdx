사용 가능한 인터페이스를 통해 [Transactions](/concepts/transactions.mdx), [Checkpoints](/concepts/cryptography/system/checkpoint-verification.mdx), [Objects](/concepts/object-model.mdx), [Events](/guides/developer/sui-101/using-events.mdx) 등과 같은 Sui 네트워크 데이터에 접근할 수 있다. 이 데이터는 애플리케이션 워크플로에 사용하거나 관심 있는 애플리케이션 또는 프로토콜 전반의 네트워크 동작을 분석하거나 네트워크의 일부 또는 전체를 감사하는 데 사용할 수 있다.

이 문서는 현재 Sui 네트워크 데이터에 접근하기 위해 사용할 수 있는 인터페이스를 개괄하고 이것이 점진적으로 어떻게 진화하고 있는지에 대한 개요를 함께 제공한다. 이 문서에서 언급하는 릴리스 단계는 다음 정의를 참조하라:

- **Alpha** - 변경될 수 있는 실험적 릴리스이며 프로덕션 사용에는 권장되지 않는다. 프로덕션이 아닌 환경에서 탐색용으로 사용할 수 있다.
- **Beta** - 사용자 피드백에 따라 변경될 수 있는 비교적 안정적인 릴리스이다. 프로덕션이 아닌 환경에서 테스트와 프로덕션 준비에 사용할 수 있다. 프로덕션에서 사용한다면 그 책임은 사용자에게 있다. 프로덕션이 아닌 환경에서 원하는 기능, 성능, 기타 관련 특성을 검증했고 변경에 맞춰 애플리케이션을 정기적으로 업데이트하는 데 무리가 없을 때만 사용을 고려한다.
- **Generally available (GA)** - 프로덕션에서 사용할 수 있는 완전히 안정적인 릴리스이다. 호환성이 깨지는 변경 사항은 사전에 공지된다.

## Current data access interfaces

현재는 다음 메커니즘 중 어느 것이든 사용하여 Sui 네트워크 데이터에 접근할 수 있다:

- [RPC providers](https://sui.io/developers#dev-tools)(`RPC`로 필터링한다) 또는 [Data indexer operators](https://github.com/sui-foundation/awesome-sui?tab=readme-ov-file#indexers--data-services)가 운영하는 Sui [Full nodes](guides/operator/sui-full-node.mdx)에 호스팅된 [JSON-RPC](/references/sui-api.mdx)에 직접 연결한다.
  - [Mainnet](https://fullnode.mainnet.sui.io:443), [Testnet](https://fullnode.testnet.sui.io:443), 또는 [Devnet](https://fullnode.devnet.sui.io:443) 로드 밸런서 URL은 Sui Foundation이 관리하는 풀 노드를 추상화한다. 이는 프로덕션 사용에 권장되지 않는다.
- 관심 있는 데이터를 Postgres 데이터베이스에 지속적으로 적재하기 위해 자체 [custom indexer](/guides/developer/advanced/custom-indexer.mdx)를 설정한다.

:::info

alpha 또는 beta로 제공되는 [future-oriented interfaces](#future-data-access-interfaces) 중 하나를 사용할 수도 있지만 이는 프로덕션 사용에 권장되지 않는다.

:::

### JSON-RPC

현재는 Sui 풀 노드에서 실시간 데이터나 과거 데이터를 얻을 수 있다. 과거 데이터의 리텐션 기간은 노드 운영자가 구현하는 [pruning strategy](/guides/operator/data-management.mdx#sui-full-node-pruning-policies)에 따라 달라지지만 현재 모든 풀 노드의 기본 구성은 과거 transaction 데이터에 대해 암묵적으로 [Sui Foundation-managed key-value store](/guides/operator/data-management.mdx#sui-full-node-key-value-store-backup)로 폴백한다.

:::caution

WebSocket 기반 JSON RPC인 `suix_subscribeEvent`와 `suix_subscribeTransaction`은 2024년 7월에 deprecated되었다. 애플리케이션에서 이 RPC들에 의존하지 말아야 하며 향후 대안에 대해서는 [Future data access interfaces](#future-state-data-interfaces)를 참조하라.

:::

### Custom indexer

애플리케이션에 필요한 데이터의 타입, 세분성, 리텐션 기간을 더 세밀하게 제어해야 하거나 관계형 데이터베이스가 더 적합한 특정 쿼리 패턴이 있다면 자체 [custom indexer](/guides/developer/advanced/custom-indexer.mdx)를 설정하거나 이미 이를 설정했을 수도 있는 [Data indexer operator](https://github.com/sui-foundation/awesome-sui?tab=readme-ov-file#indexers--data-services)에 문의할 수 있다.

자체 인덱서를 설정하면 그 지속적인 유지보수와 관련 인프라 및 운영 비용은 사용자 책임이다. 애플리케이션의 리텐션 요구 사항을 고려해 관계형 데이터베이스에 pruning strategy를 구현함으로써 비용을 줄일 수 있다.

:::info

custom indexer는 앱 또는 프로토콜별 로직과 데이터 레이아웃에 적합한 선택지이다. 또한 이는 [future data access interfaces](#future-data-access-interfaces)의 일부이기도 하다.

:::

### gRPC API

앞서 언급했듯이 [gRPC API](../../../concepts/grpc-overview.mdx)는 풀 노드의 JSON-RPC를 대체하며 따라서 gRPC API가 Generally available가 되면 JSON-RPC는 deprecated된다. 두 인터페이스의 메시지 및 요청 형식 차이 외에도 gRPC API에는 몇 가지 핵심적인 기능 차이가 있다:

- 레코드를 폴링하지 않고도 애플리케이션에서 실시간 스트리밍 데이터를 소비할 수 있도록 streaming 또는 subscription API 엔드포인트를 사용한다. 이 지원은 JSON-RPC에서 deprecated된 WebSocket 지원을 대체한다.
- 앞서 언급한 과거 transaction 데이터용 [Sui Foundation-managed key-value store](/guides/operator/data-management.mdx#sui-full-node-key-value-store-backup)로의 암묵적 폴백은 없다. 풀 노드 운영자, RPC provider, data indexer operator는 과거 데이터를 가져오기 위한 명시적 의존성으로 사용할 수 있는 유사한 아카이브 스토어의 자체 인스턴스를 운영하는 것이 권장된다. 자세한 내용은 [Archival Store and Service](#archival-store-and-service)를 참조하라.

GraphQL RPC와의 비교는 [When to use gRPC vs GraphQL with General-purpose Indexer](#when-to-use-grpc-vs-graphql-with-general-purpose-indexer)를 참조하라.

:::info

[gRPC API](../../../concepts/grpc-overview.mdx)는 beta이며 사용자 피드백에 따라 변경될 수 있는 비교적 안정적인 릴리스이다. 프로덕션이 아닌 환경에서 테스트와 프로덕션 준비에 사용할 수 있다.

:::

### General-purpose Indexer

앞서 언급했듯이 [General-purpose Indexer](../../../concepts/graphql-indexer.mdx)에는 [custom indexing framework](../../../concepts/custom-indexing-framework.mdx)의 성능 좋고 확장 가능한 구현체가 포함된다. 기반 프레임워크는 remote checkpoint store와 풀 노드 RPC를 사용해 데이터를 수집한다.

General-purpose Indexer는 서로 다른 종류의 Sui 네트워크 데이터를 Postgres 관계형 테이블에 병렬로 적재하도록 매끄럽게 구성할 수 있다는 점에서 선언적이다. 이는 Postgres 호환 데이터베이스로의 데이터 수집 성능을 향상한다. 또한 Postgres 호환 데이터베이스의 서로 다른 테이블에 pruning을 구성할 수 있어 원하는 성능과 비용 특성의 조합에 맞게 튜닝할 수 있다.

:::info

General-purpose Indexer는 beta이며 사용자 피드백에 따라 변경될 수 있는 비교적 안정적인 릴리스이다. 프로덕션이 아닌 환경에서 테스트와 프로덕션 준비에 사용할 수 있다.

:::

### GraphQL RPC

[GraphQL RPC Server](../../../concepts/graphql-indexer.mdx)는 General-purpose Indexer의 Postgres 호환 데이터베이스, [Archival Service](#archival-store-and-service), 풀 노드에서 데이터를 읽는 성능 좋은 GraphQL RPC 레이어이다. GraphQL RPC는 gRPC API의 대안이다. 현재 애플리케이션에서 이미 JSON-RPC를 사용하고 있다면 General-purpose Indexer, Postgres 호환 데이터베이스, GraphQL RPC server의 결합 스택을 직접 운영하거나 RPC provider 또는 인덱서 운영자가 제공하는 서비스로 활용하여 GraphQL RPC로 마이그레이션할 수 있다.

GraphQL RPC Server는 Postgres 호환 데이터베이스의 여러 테이블에서 데이터를 결합할 수 있게 해 주는 경량 서버 구성 요소이며 이는 GraphQL의 표현력 있는 쿼리 시스템을 사용하므로 프런트엔드 개발자에게 매력적이다.

gRPC API와의 비교는 [When to use gRPC vs GraphQL with General-purpose Indexer](#when-to-use-grpc-vs-graphql-with-general-purpose-indexer)를 참조하라.

:::info

GraphQL RPC Server는 beta이며 사용자 피드백에 따라 변경될 수 있는 비교적 안정적인 릴리스이다. 프로덕션이 아닌 환경에서 테스트와 프로덕션 준비에 사용할 수 있다.

:::

### Archival Store and Service

장기적인 과거 온체인 데이터 접근은 개발자에게 필수적이지만 풀 노드는 확장성과 성능을 위해 제한된 리텐션을 적용한다. [Archival Store and Service](../../../concepts/archival-store.mdx)를 사용하면 [Bigtable](https://cloud.google.com/bigtable)과 같은 플러그형 스토리지 백엔드와 [`LedgerService` 엔드포인트와 호환되는 gRPC 인터페이스](../../../concepts/grpc-overview.mdx)를 통해 과거 온체인 데이터에 접근할 수 있다.

이 인프라는 GraphQL RPC, gRPC 기반 앱, 데이터 플랫폼을 위한 과거 데이터 백본 역할을 하며 풀 노드가 pruning한 이후에도 오래된 transaction, checkpoint, object state를 효율적으로 포인트 조회할 수 있게 해 준다.

다음과 같은 작업을 할 수 있다:

- 풀 노드를 사용할 때 누락된 데이터를 gRPC 기반 archival service를 통해 archival store에서 조회한다.
- 리텐션되지 않은 데이터를 포함하는 GraphQL RPC 쿼리를 지원한다.
- 직접 서비스를 호스팅하거나 provider를 이용하거나 속도 제한이 있는 public-good 버전을 사용한다.

자세한 내용은 [Archival Store and Service](../../../concepts/archival-store.mdx)를 참조하라.

### When to use gRPC vs GraphQL with General-purpose Indexer

다음 표의 상위 기준을 사용하여 gRPC API와 General-purpose Indexer를 사용하는 GraphQL RPC 중 어느 쪽이 사용 사례에 더 적합한지 판단할 수 있다. 이 목록은 완전하지 않으며 일부 사용 사례에는 두 옵션 모두 적절하게 동작할 수 있다.

| Dimension | gRPC API | GraphQL RPC with General-purpose Indexer |
| --- | --- | --- |
| Type of application or data consumer. | Web3 거래소, DeFi 마켓 메이커 앱, 기타 DeFi 프로토콜 또는 초저지연이 필요한 앱에 적합하다. | 웹앱 빌더 또는 지연 시간 요구가 다소 완화된 빌더에 적합하다. |
| Query patterns. | 서로 다른 엔드포인트에서 데이터를 별도로 읽고 클라이언트 측에서 결합해도 괜찮고 바이너리 형식 덕분에 직렬화, 파싱, 검증이 더 빠르다. | 단일 요청에서 여러 테이블의 데이터를 결합할 수 있어 클라이언트를 더 쉽게 분리할 수 있고 페이지네이션된 결과를 포함해 유사한 checkpoint 전반에서 서로 다른 테이블의 일관된 데이터를 반환한다. |
| Retention period requirements. | 기본 리텐션 기간은 2주이며 실제 구성은 풀 노드 운영자의 필요와 목표에 따라 달라진다. 표 아래의 이력 관련 참고를 확인하라. | Postgres 데이터베이스의 기본 리텐션 기간은 4주이며 실제 구성은 사용자 또는 RPC provider 또는 data indexer operator의 필요에 따라 달라진다. 표 아래의 이력 관련 참고를 확인하라. |
| Streaming needs. | beta 릴리스 전에 streaming 또는 subscription API가 포함된다. | Subscription API가 계획되어 있지만 GA 이후에 제공된다. |
| Incremental costs. | 이미 풀 노드 JSON-RPC를 사용하고 있다면 추가 비용이 거의 없거나 전혀 없다. | 이미 풀 노드 JSON-RPC를 사용하고 있고 리텐션 기간 및 쿼리 패턴 차이가 중요하지 않다면 추가 비용이 다소 크다. |

:::info

이 표는 두 옵션의 기본 리텐션 기간만 언급한다. 풀 노드 운영자, RPC provider, data indexer operator가 성능에 큰 영향을 주지 않고 이 값을 몇 배 더 높게 구성하는 것은 합리적일 것으로 예상된다. 또한 기본적으로 GraphQL RPC Server는 기반 Postgres 데이터베이스에 설정된 리텐션 기간을 넘는 과거 데이터에 대해 archival key-value store에 직접 연결할 수 있다. 반면 gRPC API에는 이러한 archival key-value store 직접 연결 기능이 없다.

각 옵션의 GA 릴리스 전에 관련 가이드라인이 제공될 예정이다. 여기에는 선택한 인터페이스의 구성된 리텐션 기간을 넘어서는 과거 데이터에 접근하는 방법에 대한 권장 사항이 포함된다.

:::

gRPC와 GraphQL의 일반적인 차이를 설명하는 다음 글도 참조하라. 다만 차이점의 정확성과 진위는 직접 실험하여 검증해야 한다.
