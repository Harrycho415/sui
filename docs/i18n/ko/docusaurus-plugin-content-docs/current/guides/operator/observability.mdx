우수한 observability 역량은 Sui의 개발과 성장에 핵심적이다. 여러 클라이언트와 validator 프로세스가 잠재적으로 전 세계적인 네트워크에 분산된 Sui의 분산적이고 비동기적인 특성 때문에 이는 더욱 어렵다.

Sui의 observability stack은 [Tokio tracing](https://tokio.rs/blog/2019-08-tracing) 라이브러리에 기반한다. 이 문서의 나머지 부분에서는 Sui에서 structured logging과 metrics를 통해 좋은 observability를 달성하는 구체적인 측면을 설명한다.

:::info

여기서 나오는 출력은 주로 Sui 운영자, 관리자, 개발자를 위한 것이다. 로그와 trace의 내용은 validator의 권위 있고 인증된 출력을 나타내지 않으며, 잠재적으로 비잔틴 동작의 영향을 받을 수 있다.

:::

## Contexts, scopes, and tracing transaction flow {#context-scope-trace}

Sui와 같은 분산적이고 비동기적인 시스템에서는 단일 스레드에서 시간 순으로 개별 로그만 보는 방식에 의존할 수 없다. 이 문제를 해결하려면 structured logging 접근 방식을 사용한다. Structured logging은 스레드와 프로세스 경계를 넘어 로그, event, 기능 블록을 서로 연결할 수 있는 방법을 제공한다.

### Spans and events {#spans-and-events}

[Tokio tracing](https://tokio.rs/blog/2019-08-tracing) 라이브러리에서 structured logging은 [spans and events](https://docs.rs/tracing/0.1.31/tracing/index.html#core-concepts)를 사용해 구현된다.
span은 함수 호출 하나, future, 비동기 task 등 기능의 전체 블록을 포괄한다. span은 중첩될 수 있고, span 안의 key-value pair는 함수 내부의 event나 로그에 context를 제공한다.

- span과 그 key-value pair는 transaction ID와 같이 포함된 로그에 필수 context를 추가한다.
- span은 코드의 서로 다른 구간에서 소비한 시간도 추적하므로 distributed tracing 기능을 가능하게 한다.
- 개별 로그도 파싱, 필터링, 집계를 돕기 위해 key-value pair를 추가할 수 있다.

관심을 가질 만한 context 정보는 다음과 같다:

- TX Digest
- Object references/ID, when applicable
- Address
- Certificate digest, if applicable
- For Client HTTP endpoint: route, method, status
- Epoch
- Host information, for both clients and validators

다음 다이제스트에서 `process_tx`는 초기 transaction 요청 처리를 포괄하는 span이고, `Checked locks`는 validator의 transaction 처리 메서드 안에 있는 단일 로그 메시지이다.

span 안에서 발생하는 모든 로그 메시지는 span에 정의된 key-value 속성, 즉 `tx_digest`와 추가된 다른 모든 필드를 상속한다. 로그 메시지는 자체 key와 value를 설정할 수도 있다. 로그가 span 속성을 상속한다는 점 덕분에 예를 들어 스레드와 프로세스 경계를 넘어 transaction의 흐름을 추적할 수 있다.

## Key-value pairs schema {#key-value-pairs-schema}

span은 단일 event가 아니라 전체 시간 블록을 포착하므로 start, end, duration 등을 캡처하고 분석하여 tracing, 성능 분석 등에 활용할 수 있다.

### Tags - keys {#tags-keys}

핵심 아이디어는 모든 event와 span에 key-value pair 태그를 붙이는 것이다. 어떤 context나 중첩된 context 안에서 기록되는 event도 context 수준의 태그를 상속한다.

이 태그들은 분석과 필터링의 기준이 되는 _field_ 를 나타낸다. 예를 들어 broadcasts를 걸러내고, bad stake가 특정 양을 넘었지만 error로 이어질 정도는 아닌 모든 인스턴스를 볼 수 있다.

## Logging levels {#logging-levels}

이는 항상 까다로운 문제다. 특히 기본값에서 적절한 정도의 verbosity와 이것이 고성능 시스템이라는 점 사이의 균형을 맞춰야 하기 때문이다.

| Level | 메시지 유형 |
| ----- | ----------- |
| Error | 프로세스 수준 장애(transaction 수준 errors는 제외한다. 그런 것은 매우 많을 수 있다) |
| Warn  | 비정상적이거나 비잔틴 활동 |
| Info  | 상위 수준의 집계 통계, data sync, epoch 변경과 관련된 주요 이벤트 |
| Debug | 개별 transactions에 대한 상위 수준 tracing(예: Gateway/client side -> validator -> Move execution 등) |
| Trace | 개별 transactions에 대한 극도로 상세한 tracing |
|       |             |

`Info`에서 `Debug`로 변경하면 메시지 양이 훨씬 많아진다.

전체 로깅 레벨과 개별 component의 레벨은 `RUST_LOG` 환경 변수로 설정한다. span 내부의 특정 span이나 태그만 따로 필터링하는 것도 가능하다.

자세한 내용은 [EnvFilter](https://docs.rs/tracing-subscriber/latest/tracing_subscriber/filter/struct.EnvFilter.html) 주제를 참고한다.

## Metrics {#metrics}

Sui에는 Prometheus 기반 metrics가 포함된다:

- RPC Server API metrics와 latency를 위한 `rpc_requests_by_route` 및 관련 metrics(`rpc-server.rs` 참조)
- validator transaction metrics(`authority.rs`의 `AuthorityMetrics` 참조)

## Viewing logs, traces, metrics {#viewing-logs-traces-metrics}

tracing 아키텍처는 출력을 처리하고 서로 다른 sink로 전달해 볼 수 있도록 tracing 라이브러리에 연결하는 [subscribers](https://github.com/tokio-rs/tracing#project-layout) 개념에 기반한다. 여러 subscriber를 동시에 활성화할 수 있다.

예를 들어 JSON 로그를 [Vector](https://vector.dev) 같은 로컬 sidecar 로그 포워더를 통해 전달하고, 다시 ElasticSearch 같은 대상으로 보낼 수 있다.

Vector 같은 로그 및 metrics 집계기를 사용하면 validator 서버를 중단하지 않고도 쉽게 재구성할 수 있고, observability 트래픽을 오프로드할 수도 있다.

metrics는 기본적으로 `<host>:9184/metrics`의 Prometheus scrape endpoint로 제공된다.

### Stdout (default) {#stdout-default}

기본적으로 로그(span은 제외)는 사람이 읽기 쉬운 형식으로 포맷되어 stdout으로 출력되며, 각 줄 끝에는 key-value 태그가 붙는다.

필터링을 포함한 커스텀 로깅 출력을 위해 `RUST_LOG`를 구성할 수 있다. 자세한 내용은 앞서 설명한 [Logging levels](#logging-levels) 섹션을 참고한다.

### Tracing and span output {#tracing-and-span-output}

상세한 span 시작 및 종료 로그를 생성하려면 `SUI_JSON_SPAN_LOGS` 환경 변수를 정의한다. 그러면 모든 출력이 JSON 형식이 되는데, 사람이 읽기에는 덜 적합하므로 기본적으로는 활성화되지 않는다.

### Jaeger (seeing distributed traces) {#jaeger}

중첩된 span을 [Jaeger](https://www.jaegertracing.io)에서 시각적으로 보려면 다음을 수행한다:

1. 로컬 Jaeger container를 실행하려면 다음을 실행한다:

```sh
$ docker run -d -p6831:6831/udp -p6832:6832/udp -p16686:16686 jaegertracing/all-in-one:latest
```

1. Sui를 다음과 같이 실행한다(trace는 가장 상세한 span을 활성화한다):

```sh
$ SUI_TRACING_ENABLE=1 RUST_LOG="info,sui_core=trace" ./sui start
```

1. Sui CLI client로 transfers를 몇 번 실행하거나 benchmarking tool을 실행한다.
1. 브라우저에서 `http://localhost:16686/`로 이동해 service로 Sui를 선택한다.

:::info

현재는 중첩되지 않은 별도의 span이 단일 trace로 연결되지 않는다.

:::

### Live async inspection / Tokio Console {#tokio-console}

[Tokio-console](https://github.com/tokio-rs/console)은 Tokio를 사용하는 Rust 앱을 실시간으로 분석하고 디버그를 돕기 위해 설계된 훌륭한 CLI tool이다. 🎉 특별한 subscriber에 의존한다.

1. 다음과 같이 특수 플래그를 사용해 Sui를 빌드한다: `RUSTFLAGS="--cfg tokio_unstable" cargo build`.
1. `SUI_TOKIO_CONSOLE`을 1로 설정해 Sui를 시작한다.
1. console repo를 clone한 뒤 `cargo run`으로 console을 실행한다.

:::tip

Tokio-console 지원을 추가하면 Sui validator/gateway의 속도가 크게 느려질 수 있다.

:::

### Memory profiling {#memory-profiling}

Sui는 대부분의 플랫폼에서 기본적으로 [jemalloc memory allocator](https://jemalloc.net/)를 사용하며, jemalloc의 sampling profiler를 사용해 자동 메모리 프로파일링을 활성화하는 코드가 포함되어 있다. 이는 매우 가볍고 프로덕션 사용을 염두에 두고 설계되었다. 프로파일링 코드는 기본적으로 총 메모리가 20% 증가했을 때에만 최대 5분마다 profile을 기록한다. 프로파일링 파일 이름은 `jeprof.<TIMESTAMP>.<memorysize>MB.prof` 형식이므로 디버깅을 위해 metrics와 incident를 쉽게 연관 지을 수 있다.

메모리 프로파일링이 동작하려면 환경 변수 `_RJEM_MALLOC_CONF=prof:true`를 설정해야 한다. [Docker image](https://hub.docker.com/r/mysten/sui-node)를 사용한다면 이 값은 자동으로 설정된다.

[Bytehound](https://github.com/koute/bytehound) 같은 일부 allocator 기반 heap profiler를 실행하면 `jemalloc_ctl` 통계 API를 방해하거나 구현하지 않기 때문에 자동 jemalloc 프로파일링이 사실상 비활성화된다.

profile 파일을 보려면, profile을 수집한 것과 동일한 플랫폼에서 다음을 수행해야 한다:

1. `libunwind`, graphviz의 `dot` 유틸리티, jeprof를 설치한다. Debian에서는 다음을 실행한다: `apt-get install libjemalloc-dev libunwind-dev graphviz`.
1. 디버그 심볼을 포함해 빌드한다: `cargo build --profile bench-profiling`
1. `$SUI_REPO/target/bench-profiling`로 이동한다.
1. `jeprof --svg sui-node jeprof.xxyyzz.heap`를 실행한다. 파일 이름의 timestamp와 memory size를 기준으로 heap profile을 선택한다.

:::tip

자동 메모리 프로파일링을 사용하면 앞서 나열한 것 외에 추가 환경 변수를 구성할 필요는 없다. 사용자 정의 프로파일링 옵션을 구성할 수도 있다:

- [Heap Profiling](https://github.com/jemalloc/jemalloc/wiki/Use-Case%3A-Heap-Profiling)
- [heap profiling with jemallocator](https://gist.github.com/ordian/928dc2bd45022cddd547528f64db9174)

예를 들어 `_RJEM_MALLOC_CONF`를 다음과 같이 설정할 수 있다:
`prof:true,lg_prof_interval:24,lg_prof_sample:19`

앞의 설정은 프로파일링을 켜고, 할당된 `2^19` 또는 512KB마다 sample을 수행하며, 할당된 메모리가 `2^24` 또는 16MB가 될 때마다 profile을 덤프한다는 뜻이다. 다만 자동 프로파일링은 더 나은 이름과 더 긴 간격으로 파일을 생성하도록 설계되어 있으므로 기본 구성을 재정의하는 것은 보통 권장되지 않는다.

:::
