Currency Standard는 Sui에서 화폐를 생성하기 위해 Move 스마트 컨트랙트가 사용하는 기술 표준이다. `sui::coin_registry` 모듈은 표준을 정의하는 로직을 제공하며, 통합된 메타데이터, 향상된 공급 추적, 그리고 개선된 규제 기능을 제공한다.

Sui에서의 화폐 생성 표준화는 지갑, 거래소, 그리고 다른 스마트 컨트랙트가 추가적인 처리 로직 없이 SUI를 관리하는 것과 동일하게 Sui에서 생성된 화폐를 관리할 수 있음을 의미한다.

SUI 네이티브 통화와 네트워크에서의 사용에 대해 더 알아보려면 [Sui Tokenomics](/concepts/tokenomics.mdx)를 참조한다.

Sui의 화폐는 Currency Standard를 따르면서 특화된 ability를 제공할 수 있다. 예를 들어, 제작자가 특정 address를 deny list에 추가하여 식별된 address가 transaction의 입력으로 해당 화폐를 사용할 수 없도록 하는 규제된 화폐를 생성할 수 있다. 기존 시스템과 레지스트리 시스템 모두 다양한 수준의 정교함으로 규제된 화폐를 지원한다.

## Fungible tokens

Sui에서 `Currency<T>` 타입은 개방형 대체 가능 토큰을 나타낸다(폐쇄형 토큰에 대해서는 `Token<T>` 참조). 화폐는 타입 매개변수 `T`로 명명되며, `T`는 모든 `Currency<T>` 인스턴스에 적용되는 메타데이터(이름, 심볼, 소수점 정밀도 등)와 연관된다. `sui::coin_registry` 모듈은 `Currency<T>`를 대체 가능한 것으로 취급하는 인터페이스를 제공하며, 이는 전통적인 법정 화폐가 작동하는 방식과 유사하게 `Currency<T>`의 한 인스턴스에 보관된 `T` 단위가 다른 `T` 단위와 교환 가능함을 의미한다.

#### Core components

**`CoinRegistry`:** 모든 화폐 관련 작업을 조정하는 메인 시스템 object이다. 이 shared object는 모든 레지스트리 작업의 진입점 역할을 하며 네트워크 초기화 중에 한 번 생성된다. Address는 `0xc`이다.

**`Currency<T>`:** 다음을 포함하여 각 코인 타입에 대한 포괄적인 정보를 저장하는 레지스트리 시스템의 핵심이다:
- **메타데이터 관리:** 이름, 심볼, 소수점, 설명, 아이콘 URL과 같은 기본 화폐 정보.
- **공급 추적:** 공급 상태 정보(고정, 소각 전용, 또는 알 수 없음)를 유지한다.
- **규제 상태:** 코인이 deny list capability로 규제되는지 추적한다.
- **Capability 참조:** 해당 코인 타입에 대한 `TreasuryCap` 및 `MetadataCap` 링크.
- **확장성:** 향후 개선을 위한 추가 필드를 포함한다.

#### Supply states

레지스트리는 세 가지 다른 공급 관리 모델을 지원한다:

- **고정 공급:** 총 공급량이 영구적으로 잠겨 변경할 수 없다.
- **소각 전용 공급:** 새로운 코인을 민트할 수는 없지만, 기존 코인은 소각할 수 있다.
- **비제어 공급:** `TreasuryCap` 보유자가 민팅과 소각을 제어한다.

#### Regulatory states

화폐는 다른 규제 상태를 가질 수 있다:

- **규제됨:** 화폐에 `DenyCapV2`가 연결되어 있어 address의 사용을 제한할 수 있다.
- **규제되지 않음:** 화폐가 deny list capability 없이 생성되었다.
- **알 수 없음:** 규제 상태가 결정되지 않았으며, 일반적으로 기존 마이그레이션에서 발생한다.

레지스트리는 두 가지 다른 화폐 생성 흐름을 지원한다:

- **표준 생성 (권장):**

    화폐 타입이 게시된 후 언제든지 `new_currency` 함수를 사용할 수 있다. 이 함수는 즉시 shared `Currency<T>` object를 생성한다. `T` 타입은 `public struct MyCoin has key { id: UID }`와 같이 key-only 타입이어야 한다.

    <ImportContent source="crates/sui-framework/packages/sui-framework/sources/registries/coin_registry.move" mode="code" fun="new_currency" signatureOnly noComments />

- **One-Time Witness (OTW) creation:**

    :::caution 

    새 화폐의 OTW 생성은 2단계 프로세스이다. 초기화 프로세스는 패키지 게시로 시작된다. 그 후, 화폐를 레지스트리에 배치하기 위해 `finalize_registration` 호출이 필요하다.

    :::

    `new_currency_with_otw` 함수는 고유성 증명을 위해 OTW를 사용한다. 더 자세한 내용은 The Move Book의 [One-Time Witness](https://move-book.com/programmability/one-time-witness)를 참조한다.

- **규제 적용:** Deny list capability를 추가한다.
- **공급 모델 설정:** 고정, 소각 전용, 또는 유동적 공급 중에서 선택한다.

## Regulated currency

Currency Standard는 규제된 코인 생성을 지원한다. `finalize`를 호출하기 전 초기화 단계에서 `make_regulated` 함수를 사용한다. 이 함수는 `Currency<T>`에 deny list capability를 추가하고 Coin Registry 내에서 규제 상태를 추적한다. 함수는 보유자가 deny list의 address 목록을 유지 관리할 수 있게 해주는 `DenyCap`을 반환한다.

### `DenyList` object

시스템에 의해 생성된 `DenyList` shared object는 특정 규제된 화폐를 사용할 수 없는 address를 포함한다. `DenyCap`에 대한 접근 권한이 있다면 `sui::coin` 모듈을 사용하여 사용자 목록을 유지 관리할 수 있다. Address를 추가하고 제거하려면 `coin::deny_list_v2_add` 및 `coin::deny_list_v2_remove` 함수를 사용한다.

### Global pause switch

규제된 화폐는 `allow_global_pause` 불리언 필드를 포함한다. `true`로 설정되면, 해당 화폐 코인 타입의 `DenyCapV2` object 보유자는 `coin::deny_list_v2_enable_global_pause` 함수를 사용하여 코인 활동을 무기한 일시 중지할 수 있다. 보유자가 일시 중지를 시작하는 즉시, 네트워크는 모든 transaction의 입력으로 해당 코인 타입을 허용하지 않는다. 다음 epoch(epoch는 약 24시간 지속됨)가 시작되면, 네트워크는 추가적으로 모든 address가 해당 코인 타입을 수신하는 것을 허용하지 않는다.

해당 화폐 코인 타입의 `DenyCapV2` object 보유자가 `coin::deny_list_v2_disable_global_pause`를 사용하여 일시 중지를 해제하면, 코인은 즉시 transaction 입력으로 다시 사용할 수 있게 된다. 하지만 address는 다음 epoch까지 해당 코인 타입을 수신할 수 없다.

글로벌 일시 중지 기능은 화폐의 deny list에 영향을 미치지 않는다. 화폐에 대한 일시 중지를 해제한 후에도, deny list에 포함된 모든 address는 여전히 해당 화폐의 코인과 상호 작용할 수 없다.

## `TreasuryCap`

`coin_registry::new_currency` 또는 `coin_registry::new_currency_with_otw` 함수를 사용하여 화폐를 생성할 때, 코인을 생성하는 스마트 컨트랙트의 게시자는 화폐가 확정(finalize)되면 `TreasuryCap` object를 받는다. `TreasuryCap` object는 새로운 화폐 코인을 민트하거나 현재 코인을 소각하는 데 필요하다(화폐 공급 상태에 따라 다름). 결과적으로 이 object에 접근할 수 있는 address만이 Sui 네트워크에서 화폐 공급을 유지 관리할 수 있다.

`TreasuryCap` object는 전송 가능하므로, `TreasuryCap`을 제3자에게 전송하면 생성한 화폐의 관리를 넘길 수 있다. Object를 전송한 후에는, 본인이 직접 코인을 민트하고 소각할 수 없게 된다.

## Currency metadata

화폐 메타데이터는 레지스트리의 `Currency<T>` object 내에 중앙 집중식으로 저장된다. 메타데이터 업데이트는 `MetadataCap<T>` capability에 의해 제어되며, 그 상태는 다음과 같을 수 있다:

- **클레임되지 않음:** `MetadataCap`이 아직 클레임되지 않았다.
- **클레임됨:** `MetadataCap`이 클레임되었으며 업데이트에 사용할 수 있다. 
- **삭제됨:** `MetadataCap`이 영구적으로 삭제되어 향후 업데이트가 방지된다.

### Metadata fields

메타데이터 object의 필드는 다음과 같다:

#### `CoinMetadata`

| Name          | Description                                                                                                                                                                                              |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `registry`          | `CoinRegistry` 시스템 object ID (`0x3`).                                                                                                                                                             |
| `decimals`    | 토큰이 사용하는 소수점 자리수이다. 이 필드를 `3`으로 설정하면, 값 `1000`인 토큰은 `1.000`으로 표시된다.                                                                              |
| `name`        | 코인의 이름이다.                                                                                                                                                                                        |
| `symbol`      | 코인의 심볼이다. 이는 `name`과 동일한 값일 수 있지만, 일반적으로 5자 미만의 대문자이다. 예를 들어, `SUI`는 Sui의 네이티브 코인에 대한 `symbol`이지만 `name` 또한 `SUI`다. **ASCII 인쇄 가능 문자만 허용된다**.|
| `description` | 토큰을 설명하기 위한 짧은 설명이다.                                                                                                                                                               |
| `icon_url`    | **필수 필드**. 지갑, 익스플로러, 및 기타 앱에서 표시하는 데 사용되는 토큰 아이콘의 URL이다.                                                                                                                    |

## Minting and burning coins

Coin Registry는 고급 공급 관리 모델을 지원한다:

- **알 수 없는 공급:** `TreasuryCap`을 통한 전통적인 민팅/소각.
- **고정 공급:** 총 공급량이 영구적으로 잠겨 있으며, 민팅이나 소각이 허용되지 않는다.
- **소각 전용 공급:** 새로운 민팅은 허용되지 않지만, 기존 코인은 레지스트리 함수를 통해 소각될 수 있다.

### Mint

새로운 코인을 생성하려면 `coin::mint` 함수를 사용한다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="mint" signatureOnly noComments />

서명은 `TreasuryCap`, 생성된 코인의 값, 그리고 transaction 컨텍스트로 함수를 호출하면 `Coin<T>`가 생성됨을 보여준다. 함수는 `TreasuryCap`의 총 공급량을 자동으로 업데이트한다. 표시될 때, 코인 `value`는 메타데이터의 `decimals` 값을 따른다. 따라서 `decimal` 값이 `6`인 코인 `value`로 1000000을 제공하면, 코인의 값은 `1.000000`으로 표시된다.

### Burn

소각 전용 공급 상태인 화폐의 경우, `TreasuryCap` 없이 `coin_registry::burn` 및 `coin_registry::burn_balance` 함수를 사용하여 레지스트리를 통해 직접 코인을 소각할 수 있다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/registries/coin_registry.move" mode="code" fun="burn,burn_balance" signatureOnly noComments />

알 수 없는 공급 상태인 화폐의 경우, `coin::burn` 함수를 사용하여 현재 코인을 소각한다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="burn" signatureOnly noComments />

서명은 `TreasuryCap`과 소각하려는 코인 object만이 필요한 입력임을 보여주며, 공급이 감소된 양(코인의 값)을 반환한다. 이 함수는 공급량보다 더 많은 코인을 소각하도록 허용하지 않는다.

## Adding and removing addresses to and from the deny list

Deny list는 규제된 화폐에만 적용된다. 규제된 화폐를 생성할 때, 시스템이 생성한 `DenyList` object에서 address를 추가하고 제거할 권한을 부여하는 `DenyCapV2`를 받는다. 화폐 목록에 있는 모든 address는 추가되는 즉시 해당 코인을 transaction의 입력으로 사용할 수 없게 된다. Address가 deny list에 추가된 후 다음 epoch가 되면, 해당 address는 추가적으로 코인 타입을 수신할 수 없게 된다. 즉, 화폐의 deny list에 추가된 address는 즉시 코인을 보낼 수 없게 된다. 다음 epoch가 시작되면, 해당 address는 여전히 코인을 보낼 수 없을 뿐만 아니라 받을 수도 없게 된다. 그 시점부터 `DenyCapV2` 보유자가 deny list에서 명시적으로 제거할 때까지 해당 address는 화폐와 상호 작용할 수 없다.

### Add address to deny list

코인의 deny list에 제공된 address를 추가하려면 `coin::deny_list_v2_add` 함수를 사용한다. 함수의 서명은 다음과 같다:

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="deny_list_v2_add" noComments />

코인의 deny list에서 address를 제거하려면 `coin::deny_list_v2_remove` 함수를 사용한다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="deny_list_v2_remove" noComments />

이 함수를 사용할 때, `DenyList` object (`0x403`), 코인 생성 시 받은 `DenyCapV2`, 목록에서 제거할 address, 그리고 transaction 컨텍스트를 제공한다. 목록에 없는 address를 제거하려고 하면 `ENotFrozen` 오류가 발생하고 함수가 중단된다. 이 함수를 호출한 후, 제공한 address는 다음 epoch까지 코인을 사용할 수 있게 된다.

## Globally pausing and unpausing regulated coin activity

코인 활동의 글로벌 일시 중지는 규제된 코인 타입에만 적용된다.

### Pause coin activity

`allow_global_pause` 필드가 `true`로 설정된 규제된 코인 타입에 대해 네트워크 전체의 활동을 일시 중지하려면, `coin::deny_list_v2_enable_global_pause`를 사용한다. 일시 중지를 시작하려면 코인 타입의 `DenyCapV2` object를 제공해야 한다. Transaction 활동은 즉시 일시 중지되며, 일시 중지 호출 다음 epoch에는 어떤 address도 코인을 받을 수 없다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="deny_list_v2_enable_global_pause" noComments signatureOnly />

### Unpause coin activity

일시 중지된 규제된 코인의 네트워크 활동을 다시 시작하려면, `coin::deny_list_v2_disable_global_pause` 함수를 사용한다. 일시 중지와 마찬가지로, 코인 타입의 `DenyCapV2` object를 제공해야 한다. Transaction 활동은 즉시 재개되며, 일시 중지 해제 호출 다음 epoch부터 address들이 코인을 받기 시작할 수 있다.

<ImportContent source="crates/sui-framework/packages/sui-framework/sources/coin.move" mode="code" fun="deny_list_v2_disable_global_pause" noComments signatureOnly />

## Migration from Coin to Currency Standard

Currency Standard는 레거시 Coin Standard를 대체한다. Sui는 하위 호환성을 유지하면서 `CoinMetadata<T>` 시스템에서의 마이그레이션 경로를 제공한다.

마이그레이션 시스템은 데이터 무결성을 유지하고 기존 기능을 보존하기 위해 특정 제약 조건을 가지고 설계되었다. 마이그레이션은 참조에 의해 수행될 때만 무허가로 발생할 수 있으며, 이는 원본 `CoinMetadata` object가 손상되지 않은 상태로 유지되는 동안 해당 데이터가 복사되어 레지스트리에 새 `Currency` 항목을 생성함을 의미한다. 이 접근 방식은 새로운 화폐 데이터의 안전한 등록과 기존 화폐 데이터의 업데이트를 허용하지만, `MetadataCap`이 아직 클레임되지 않은 경우에만 가능하다.

그러나 시스템은 마이그레이션 중에 원본 `CoinMetadata` object가 소비되거나 파괴되는 값에 의한 무허가 마이그레이션을 허용할 수 없다. 이 제한은 일부 코인이 `CoinMetadata` 업데이트를 제어하는 거버넌스 메커니즘을 가지고 있기 때문에 존재한다. 값 기반 마이그레이션을 허용하면 거버넌스 시스템이 관리해야 할 메타데이터 object가 파괴되어 기존 거버넌스 워크플로가 돌이킬 수 없이 중단된다.

레거시 `CoinMetadata` object의 파괴는 해당 `MetadataCap`이 클레임된 후에만 허용되며, 이는 화폐 소유자가 새로운 레지스트리 시스템을 통해 제어권을 가졌다는 증거 역할을 한다. 이는 거버넌스 시스템이 여전히 레거시 메타데이터에 의존하는 동안 실수로 파괴되지 않도록 보장하며, 레거시 object를 정리하기 전에 소유자가 명시적으로 제어권을 클레임해야 하는 명확한 전환 경로를 제공한다.

이 설계는 하위 호환성을 유지하면서도 소유자가 새로운 시스템으로 완전히 마이그레이션할 준비가 될 때까지 기존 거버넌스 메커니즘을 보호하는 중앙화된 레지스트리 시스템으로의 원활한 전환을 가능하게 한다.

Coin Registry로 마이그레이션하면 다음과 같은 이점이 있다:

- **중앙 집중식 관리:** 모든 코인 정보에 대한 단일 source of truth.
- **향상된 기능:** 고급 공급 모델 및 규제 추적에 대한 액세스.
- **생태계 통합:** 지갑, 거래소, 및 dApp에 대한 더 나은 지원.
- **미래 대비:** 지속적인 레지스트리 개선에 대한 액세스.

## Best practices

### For currency creators

- **OTW 생성 사용:** 더 나은 고유성 보장을 위해 OTW 기반 생성을 선호한다.
- **초기 공급 모델 설정:** 초기화 중에 공급 모델(고정, 소각 전용, 또는 유동적)을 결정한다.
- **규제 고려:** Deny list capability가 필요한지 평가한다.
- **메타데이터 cap 관리:** 메타데이터 capability를 유지할지, 전송할지, 삭제할지 결정한다.

### For dApp developers

- **레지스트리 쿼리 우선:** 레거시 방법으로 돌아가기 전에 레지스트리에서 코인 정보를 확인한다.
- **마이그레이션 상태 처리:** 다양한 마이그레이션 상태에 있는 코인을 고려한다.
- **공급 모델 존중:** 다양한 공급 상태(고정, 소각 전용, 알 수 없음)의 의미를 이해한다.
- **규제 상태 확인:** 규제된 코인과 그 제한 사항을 인지한다.

### For infrastructure providers

- **레지스트리 마이그레이션 모니터링:** 새로운 코인 등록 및 업데이트를 추적한다.
- **공급 변경 인덱싱:** 소각 전용 코인에 대한 소각 이벤트를 모니터링한다.
- **레거시 코인 처리:** 레지스트리 및 레거시 메타데이터 시스템을 모두 지원한다.
- **효율적인 캐싱:** 레지스트리 데이터는 자주 변경되지 않으므로 캐시할 수 있다.

### Security considerations

**Capability security:**
- **`MetadataCap`:** 메타데이터 capability는 코인 브랜딩을 제어하므로 보호한다.
- **`TreasuryCap`:** Treasury capability는 민팅/소각 권한을 결정한다.
- **`DenyCapV2`:** Deny capability는 코인 사용을 제한할 수 있다.

**Validation:**
레지스트리는 몇 가지 중요한 유효성 검사를 수행한다:
- **심볼 유효성 검사:** 심볼은 ASCII 인쇄 가능 문자여야 한다.
- **고유성:** 각 코인 타입은 한 번만 등록할 수 있다.
- **공급 일관성:** 공급 상태를 다운그레이드할 수 없다.
- **권한 확인:** 적절한 capability 보유자만 변경할 수 있다.

**Migration safety:**
- **일회성 마이그레이션:** 레거시 메타데이터는 한 번만 마이그레이션할 수 있다.
- **Capability 증명:** 메타데이터 삭제에는 capability 소유권이 필요하다.
- **상태 일관성:** 규제 상태 마이그레이션은 중복 설정을 방지한다.
