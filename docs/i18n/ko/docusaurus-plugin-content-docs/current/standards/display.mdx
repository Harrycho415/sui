Sui Object Display 표준은 타입에 대한 오프체인 표현(display)의 온체인 관리를 가능하게 하는 템플릿 엔진이다. 이를 통해 object 데이터를 템플릿 문자열로 대체할 수 있다. 이 표준은 설정할 수 있는 필드를 제한하지 않는다.`{property}` 구문을 사용하여 모든 object 속성에 액세스하고 템플릿 문자열의 일부로 삽입할 수 있다.

소유한 `Publisher` object를 사용하여 타입에 대한 `sui::display`를 설정한다. `Publisher` object에 대한 자세한 내용은, *Sui Move by Example*의 [Publisher](https://examples.sui.io/basics/publisher.html) 주제를 참조한다.

Sui Move에서, `Display<T>`는 타입 `T`에 대한 명명된 템플릿 집합을 지정하는 object를 나타낸다. 예를 들어, 타입 `0x2::capy::Capy`의 경우 display 구문은: `Display<0x2::capy::Capy>` 이다.

Sui 풀 노드는 `Display` 정의와 일치시켜 타입 `T`의 모든 object를 처리하고, 쿼리에 `{ showDisplay: true }` 설정으로 object를 쿼리할 때 처리된 결과를 반환한다.

## Display properties

제안되는 기본 속성 집합은 다음과 같다:

- `name`: object의 이름. 이름은 사용자가 object를 볼 때 표시된다.
- `description`: object에 대한 설명. 설명은 사용자가 object를 볼 때 표시된다.
- `link`: 애플리케이션에서 사용할 object에 대한 링크.
- `image_url`: object에 대한 이미지가 있는 URL 또는 blob.
- `thumbnail_url`: 지갑, 익스플로러 및 기타 제품에서 미리보기로 사용할 **더 작은** 이미지에 대한 URL.
- `project_url`: object 또는 생성자와 연관된 웹사이트에 대한 링크.
- `creator`: object 생성자를 나타내는 문자열.

### Example: Sui Hero module

다음 코드 샘플은 예시 `Hero` 모듈의 `Display`가 타입 `Hero`의 `name`, `id`, `image_url` 속성에 따라 어떻게 달라지는지 보여준다. 다음은 `init` 함수가 정의하는 템플릿을 나타낸다:

```json
{
    "name": "{name}",
    "link": "https://sui-heroes.io/hero/{id}",
    "image_url": "https://sui-heroes.io/hero/{image_url}",
    "description": "A true Hero of the Sui ecosystem!",
    "project_url": "https://sui-heroes.io",
    "creator": "Unknown Sui Fan"
}
```

```move
/// 무제한 "Sui Hero" 컬렉션의 예시 - 누구나 가능
/// 자신의 Hero를 민트할 수 있다.
/// Publisher를 초기화하는 방법과 이를 사용하여 생태계에 대한 타입을
/// 설명하는 방법인 `Display<Hero>` object를 가져오는 방법을 보여준다.
module examples::my_hero;

use std::string::String;

// 생성 번들: 이 두 패키지는 종종 함께 사용된다.
use sui::package;
use sui::display;

/// Hero - 디지털 아트의 뛰어난 컬렉션으로 정의.
public struct Hero has key, store {
    id: UID,
    name: String,
    image_url: String,
}

/// 모듈을 위한 One-Time-Witness.
public struct MY_HERO has drop {}

/// 모듈 초기화 함수에서 `Publisher` object를 클레임하여
/// `Display`를 생성한다.
/// `Display`는 필드 집합으로 초기화되며(나중에 수정 가능)
/// `update_version` 호출을 통해 게시된다.
///
/// 키와 값은 초기화 함수에서 설정되지만 `Publisher` object가
/// 생성된 경우 게시 후에도 설정할 수 있다.
fun init(otw: MY_HERO, ctx: &mut TxContext) {
    let keys = vector[
        b"name".to_string(),
        b"link".to_string(),
        b"image_url".to_string(),
        b"description".to_string(),
        b"project_url".to_string(),
        b"creator".to_string(),
    ];

    let values = vector[
        // `name`의 경우 `Hero.name` 속성을 사용할 수 있고
        b"{name}".to_string(),
        // `link`의 경우 `id` 속성을 사용하여 URL을 구축할 수 있으며
        b"https://sui-heroes.io/hero/{id}".to_string(),
        // `image_url`의 경우 IPFS 템플릿 + `image_url` 속성을 사용한다.
        b"ipfs://{image_url}".to_string(),
        // 설명은 모든 `Hero` object에 대해 정적이다.
        b"A true Hero of the Sui ecosystem!".to_string(),
        // 프로젝트 URL은 일반적으로 정적이며
        b"https://sui-heroes.io".to_string(),
        // 생성자 필드는 무엇이든 될 수 있다
        b"Unknown Sui Fan".to_string(),
    ];

    // 패키지에 대한 `Publisher`를 클레임한다!
    let publisher = package::claim(otw, ctx);

    // `Hero` 타입에 대한 새로운 `Display` object를 가져온다.
    let mut display = display::new_with_fields<Hero>(
        &publisher, keys, values, ctx
    );

    // 변경 사항을 적용하기 위해 `Display`의 첫 번째 버전을 커밋한다.
    display.update_version();

    transfer::public_transfer(publisher, ctx.sender());
    transfer::public_transfer(display, ctx.sender());
}

/// 누구나 자신의 `Hero`를 민트할 수 있다!
public fun mint(name: String, image_url: String, ctx: &mut TxContext): Hero {
    Hero {
        id: object::new(ctx),
        name,
        image_url
    }
}
```

## Work with Object Display

`display::new<T>` 호출은 사용자 정의 함수나 모듈 초기화 함수에서, 또는 프로그래밍 가능한 transaction의 일부로 `Display`를 생성한다. 다음 코드 샘플은 `Display`를 생성하는 방법을 보여준다:

```move
module sui::display;

/// `T`에 대한 새로운 Display object를 가져온다.
/// Publisher는 T의 게시자여야 하며, `from_package`
/// 검사가 수행된다.
public fun new<T>(pub: &Publisher): Display<T> { /* ... */ }
```

`Display`를 생성한 후에는 수정할 수 있다. 다음 코드 샘플은 `Display`를 수정하는 방법을 보여준다:

```move
module sui::display;

/// 한 번에 여러 필드를 설정한다
public fun add_multiple(
    self: &mut Display,
    keys: vector<String>,
    values: vector<String>
) { /* ... */ }

/// 단일 필드를 편집한다
public fun edit(self: &mut Display, key: String, value: String) { /* ... */ }

/// Display에서 키를 제거한다
public fun remove(self: &mut Display, key: String ) { /* ... */ }
```

다음으로, `update_version` 호출은 변경 사항을 적용하고 이벤트를 방출하여 `T`에 대한 `Display`를 설정한다. 풀 노드는 이벤트를 수신하고 이벤트의 데이터를 사용하여 타입에 대한 템플릿을 검색한다.

다음 코드 샘플은 `update_version` 호출을 사용하는 방법을 보여준다:

```move
module sui::display;

/// Display 버전을 업데이트하고 이벤트를 방출한다
public fun update_version(self: &mut Display) { /* ... */ }
```

## Sui utility objects

Sui에서, 유틸리티 object는 capability에 대한 권한 부여를 가능하게 한다. 거의 모든 모듈에는 필요한 capability로만 액세스할 수 있는 기능이 있다. 일반 모듈은 마켓플레이스와 같이, 애플리케이션당 하나의 capability를 허용한다. 일부 capability는 온체인에서 공유 object의 소유권을 표시하거나, 다른 계정의 공유 데이터에 액세스한다. Capability를 사용하면, 사용자 인터페이스 구현을 용이하게 하기 위해 object에 대한 의미 있는 설명을 제공하는 것이 중요하다. 이는 object가 비슷할 때 잘못된 object를 실수로 전송하는 것을 방지하는 데 도움이 된다. 또한 사용자가 보는 항목에 대한 사용자 친화적인 설명을 제공한다.

다음 예시는 capy capability를 생성하는 방법을 보여준다:

```move
module capy::utility;

/// Capy 매니저에게 새로운 유전자를 추가하고 Capy 마켓을 관리할 수 있는
/// 권한을 부여하는 capability
public struct CapyManagerCap has key, store { id: UID }
```

## Typical objects with data duplication

게임 내 아이템의 일반적인 경우는 어떤 기준에 따라 그룹화된 많은 수의 유사한 object를 갖는 것이다. 크기와 민트 및 업데이트 비용을 최적화하는 것이 중요하다. 일반적으로, 게임은 그룹 또는 아이템 기준당 단일 소스 이미지 또는 URL을 사용한다. 모든 object 내부에 소스 이미지를 저장하는 것은 효율적이지 않다. 경우에 따라, 사용자는 게임에서 허용할 때 또는 게임 내 아이템을 구매할 때 게임 내 아이템을 민트한다. 이를 가능하게 하려면, 일부 IPFS/Arweave 메타데이터를 미리 생성하고 저장해야 한다. 이를 위해서는 일반적으로 아이템의 게임 내 속성과 관련이 없는 추가 로직이 필요하다.

다음 예시는 Capy를 생성하는 방법을 보여준다:

```move
module capy::capy_items;

use std::string::String;

/// 착용 가능한 Capy 아이템. 일부 아이템의 경우 무제한
/// 공급이 있을 수 있다. 그리고 같은 이름의 아이템은 동일하다.
public struct CapyItem has key, store {
    id: UID,
    name: String
}
```

## Unique objects with dynamic representation

Sui Capy는 동적 이미지 생성을 사용한다. Capy가 태어나면, 색상이나 패턴과 같은 속성이, Capy의 외관을 결정한다. 사용자가 Capy에 아이템을 장착하면, Capy의 외관이 변경된다. 사용자가 Capy에 여러 아이템을 장착하면, 아이템 조합에 대한 보너스 기회가 있다.

이를 구현하기 위해, Capy 게임 API 서비스는 사용자 주도 변경에 대한 응답으로 이미지를 새로 고친다. Capy의 URL은 `capy.id`가 있는 템플릿이다. 그러나 전체 URL과 다양한 개체군으로 인한 Capy object의 다른 필드를 저장하면 사용자가 초과 스토리지 및 증가된 가스 수수료를 지불하게 된다.

다음 예시는 동적 이미지 생성을 구현하는 방법을 보여준다:

```move
module capy::capy;

/// Capy - 유전자 조합이 다른 매우 다양한 object.
/// 동적으로 생성 + 이미지의 경우 동적 SVG
/// 생성이 사용된다.
public struct Capy has key, store {
id: UID,
genes: vector<u8>
}
```

## Objects with unique static content

이것은 가장 간단한 시나리오로 - object 자체가 모든 것을 나타낸다. 이러한 종류의 object에 메타데이터 표준을 적용하는 것은 매우 쉬우며, 특히 object가 영원히 불변으로 유지되는 경우 더욱 그렇다. 그러나, 메타데이터 표준이 발전하고 일부 생태계 프로젝트가 일부 속성에 새로운 기능을 추가하는 경우, 이 object는 항상 원래 형태로 유지되며 하위 호환성 변경이 필요할 수 있다.

```move
module sui::devnet_nft;

use std::string::String;

/// 정적 데이터를 가진 수집품. URL, name, description은
/// 민트 이벤트에서 한 번만 설정된다
public struct DevNetNFT has key, store {
id: UID,
name: String,
description: String,
url: String,
}
```
